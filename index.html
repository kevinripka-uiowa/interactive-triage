Interactive publishing triage

How should this interactive get into the story?

Use this guided decision tree to choose the least complicated workflow that still gives you enough control. The goal is not to code everything. The goal is to understand the tradeoff between convenience, constraint, control, and responsibility.

As control rises, responsibility usually rises too: assets, paths, optimization, testing, and troubleshooting.

Reference: the triage ladder

Start at the top. Move down only when the easier option cannot do the job.

WorkflowUse whenHow it gets into the storyAsset responsibilityMain risk
Native builder block The host platform already has a block that does the job. Add the block directly inside Shorthand, Tilda, Wix, Webflow, etc. Mostly handled by the builder, but still use reasonable file sizes. Convenient but constrained.
Embedded external tool A tool like Flourish, Datawrapper, Genially, or Knight Lab can make the interactive. Publish in that tool, copy embed code, paste into the story platform. Mostly handled by the external tool; check mobile and iframe sizing. The embed is a separate webpage inside your page.
Custom code inside builder You need a small custom widget and the builder allows HTML/CSS/JS. Paste code into an HTML/embed/custom-code block. Mixed. Assets often need public URLs or must be uploaded to the builder. Builders can restrict JavaScript or cause CSS conflicts.
Custom hosted interactive The interactive needs its own HTML, CSS, JS, assets, data, or libraries. Host on GitHub Pages/Netlify/Vercel, then iframe it into the story. You handle optimization, file paths, folders, and testing. More control, more things that can break.
Full custom site The entire story needs custom structure and the builder would get in the way. The story itself is an HTML/CSS/JS site hosted as files. You handle nearly everything. Highest control and highest responsibility.
